# ShipSafe – Agentic Multi-Agent Orchestration Blueprint

## 1. System Overview

ShipSafe is an **event-driven security orchestrator** designed to automatically detect, verify, and remediate vulnerabilities in code pushes.

The system uses a **stateful LangGraph workflow** combined with **Retrieval-Augmented Generation (RAG)** to provide agents with architectural context beyond the immediate code diff.

### Core Architecture Components

* **FastAPI** – Webhook ingestion and orchestration host
* **LangGraph** – Stateful agent workflow engine
* **ChromaDB** – Vector database for contextual code retrieval
* **LLM Agents** – Specialized models for detection, auditing, and remediation
* **GitHub API** – Posts remediation suggestions back to pull requests

---

## 2. Shared State (AgentState)

All agents interact through a centralized **TypedDict state object**.

This shared state allows agents to read, update, and pass information through the workflow.

```python
AgentState = TypedDict(
    "AgentState",
    {
        "raw_diff": str,
        "file_path": str,
        "context_chunks": List[str],
        "vulnerabilities": List[dict],
        "is_verified": bool,
        "remediation_patch": str,
        "audit_feedback": str,
    }
)
```

### State Fields

| Field             | Description                                                        |
| ----------------- | ------------------------------------------------------------------ |
| raw_diff          | The original git diff received from the GitHub webhook             |
| file_path         | Path of the file currently being analyzed                          |
| context_chunks    | Semantically similar code snippets retrieved from ChromaDB         |
| vulnerabilities   | List of detected security issues with metadata                     |
| is_verified       | Boolean indicating whether the Auditor confirmed the vulnerability |
| remediation_patch | Unified Diff patch generated by the Remediator                     |
| audit_feedback    | Critique provided by the Auditor if remediation fails              |

### Example Vulnerability Structure

```python
{
  "type": "SQL Injection",
  "line_number": 42,
  "description": "User input concatenated into SQL query",
  "confidence_score": 0.91
}
```

## 3. Agent Definitions

### Agent A — Detector

**Model:** Claude 4.5 Sonnet

**Goal**

Identify potential security vulnerabilities introduced by the code change.

**Input**

* raw_diff
* context_chunks

**Logic**

The Detector analyzes the incoming diff while considering surrounding architecture retrieved from ChromaDB.

Example reasoning:

* Missing authentication middleware
* Unsanitized database queries
* User-controlled input flows
* Privilege escalation risks

**Vulnerabilities Detected**

* IDOR (Insecure Direct Object Reference)
* SQL Injection
* Cross-Site Scripting (XSS)
* Broken Authentication
* Authorization bypass

**Output**

Updates the `vulnerabilities` field in the shared state.

---

### Agent B — Auditor

**Model:** CodeBERT / Security-specialized LLM

**Goal**

Act as a **zero-trust validator** to confirm whether the Detector's findings are legitimate.

**Input**

* vulnerabilities
* original_code

**Logic**

The Auditor performs high-precision validation by analyzing:

* Data flow
* Security control presence
* Framework protections
* Authorization patterns

This step dramatically **reduces false positives**.

**Output**

If vulnerability is confirmed:

```
is_verified = True
```

If rejected:

```
is_verified = False
audit_feedback = "Reason vulnerability is invalid"
```

---

### Agent C — Remediator

**Model:** Claude 4.5 Sonnet

**Goal**

Generate a secure code fix that preserves the existing architecture.

**Input**

* vulnerabilities
* context_chunks

**Logic**

The Remediator generates a patch that:

* Fixes the vulnerability
* Maintains project conventions
* Avoids breaking existing functionality

**Output**

Produces a **Unified Diff** stored in:

```
remediation_patch
```

**Example**

```diff
- query = f"SELECT * FROM users WHERE id = {user_id}"
+ query = "SELECT * FROM users WHERE id = %s"
+ cursor.execute(query, (user_id,))
```

## 4. LangGraph Workflow

The orchestration follows a **stateful directed graph**.

```
START
  ↓
Ingestion
  ↓
Chroma Retrieval
  ↓
Detection
  ↓
Audit
  ↓
Remediation
  ↓
Verification Loop
  ↓
GitHub Comment
```

### Step 1 — Ingestion

FastAPI receives a **GitHub webhook** containing the commit diff.

Actions:

* Parse diff
* Extract changed files
* Initialize AgentState

---

### Step 2 — Context Retrieval

Before detection begins:

1. Code diff is embedded
2. ChromaDB performs similarity search
3. Relevant architectural code is retrieved

This provides **architectural awareness** for agents.

Output stored in:

```
context_chunks
```

---

### Step 3 — Detection Node

Detector agent analyzes the diff.

Decision logic:

```python
if vulnerabilities == []:
    END
else:
    → Audit Node
```

---

### Step 4 — Audit Node

Auditor verifies findings.

Decision logic:

```python
if is_verified:
    → Remediation Node
else:
    → END
```

---

### Step 5 — Remediation Node

Remediator generates a secure fix.

Output:

```
remediation_patch
```

---

## 5. Self-Correction Loop

After generating a patch, the **Auditor re-validates the fix**.

Loop logic:

```
Auditor reviews patch
```

If patch is insecure:

```
iteration_count += 1
append audit_feedback
loop → Remediator
```

If patch is secure:

```
END
```

This creates a **closed-loop autonomous repair system**.

## 6. External Integrations

### FastAPI

Responsibilities:

* Webhook ingestion
* LangGraph orchestration
* State initialization

Entry endpoint example:

```
POST /webhook/github
```

---

### ChromaDB

Used for **semantic code retrieval**.

Retrieval flow:

```
diff → embedding → similarity search → context_chunks
```

This allows agents to reason with **architectural context beyond the diff**.

---

### GitHub API

Final step posts remediation suggestions directly to the pull request.

Example action:

```
POST /repos/{owner}/{repo}/issues/{pr_number}/comments
```

The comment contains:

* Vulnerability explanation
* Generated patch
* Security reasoning
